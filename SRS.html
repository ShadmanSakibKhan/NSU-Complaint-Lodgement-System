1. Introduction

1.1 Purpose

The purpose of this document is to present a detailed description of the Class Insight. It will explain the purpose and features of the system, the interfaces of the system, what the system will do, the constraints under which it must operate and how the system will react to external stimuli. This document is intended for both the stakeholders and the developers of the system and will be proposed to Nabeel Mohammed for its approval.

1.2 Document Conventions

The document is styled according to IEEE SRS template. For Heading we have used Times New Roman at 18 and 14 for Chapter and Topic respectively and have used Times New Roman at 12 for the body.

1.3 Intended Audience and Reading Suggestions

This document is intended for Faculty, Non Faculty staff, helping (gaurds, cleaners), Students, Employee, System Admin.

1.4 Product Scope

The software system will be lodge complian from Faculty, Non Faculty staff, helping (gaurds, cleaners), Students, Employee, System Admin. This system provides tools for Faculty and students to keep track of their complain system. Admin, Faculty and complian reviewer can make review complains.

1.5 References

Google.com

2. Overall Description

2.1 Product Perspective

The system was designed to aid teachers keep interaction between students on their complain of any activities . This system can be used to lodge and review of complains.

2.2 Product Functions

The major functions of the system:

  • Faculty, Non Faculty staff, helping (gaurds, cleaners), Students, Employee, System Admin can create accounts.
  • Users can sign in their account
  • Anyone can be lodge complain
  • When they complain, that will be rviewed
  • They can write complain
  • Able to do audio complain
  • Can edit text complain
  • The system will help to audio to text conversation.

2.3 User Classes and Characteristics

There are six user classes:

  • 1. System Administrators:
    • Have full access to databases and codes.
    • Can create Users.
    • Close any complain.
    • Edit any complain.
  • 2. Faculty:
    • Access Faculty Homepage.
    • Create account and sign in.
    • Lodge Complain.
    • Review Complain.
  • 3. Admin Staff:
    • Access Admin Staff Homepage
    • sign up and sign in.
    • Lodge Complain.
    • Review Complain.
  • 4. Non Faculty Staff:
    • Create account or sign in
    • Lodge Complain
  • 5. Helping Staff:
    • Create account or sign in
    • Lodge Complain
  • 6. Student:
    • Create account and sign in.
    • Lodge Complain.

    3. External Interface Requirements

    3.1 User Interfaces

    A login screen will be shown as a user open the software. There the user will able to scan his/her ID card by clicking on a button, another button will allow the user to enter the information manually. After login, user will see the report section. After clicking on the comment section user can create a complaint or any comments. If they want they also can report a comment.

    3.2 Hardware Interfaces

    The most important hardware for this complaint lodging app requires smartphone and laptop. A clear microphone or headphone should be required for audio interactions. The secondary requirement is an active Internet connection, either by mobile data or Wi-Fi. This is to communicate with the Google Firebase Server and to make API calls.

    3.3 Software Interfaces

    Native development kit will be used to develop the software. For face and eye detection, local facial detection library or a third-party online API will be implemented. To make the app fast and stable, MongoDB database will be used to store the data and sync properly through APIs. A PDF renderer will be implemented that will let the user to open a large file.

    3.4 Communications Interfaces

    The software will use APIs to create user account. During the reading session, app will take complains and send it to the server through API. During the complain recording session, app will also send the current page, time to server for statistics.




    4. System Features

    This section includes the requirements that describe all the essential functions of the system.

    The naming convention will follow the following conventions

    1. Log In

    • Description and Priority: Faculty, Students, admin staff, non faculty teaching staff, helping (guards, cleaners), system Admin should be able to sign up to the system.
    • Response Sequences: User goes to landing page URL. User clicks on sign up button. User is redirected to the create new profile form. User clicks sign up and is redirected to the homepage. System stores the data given by user for creating report.

    2. Sign Up

    • Description and Priority: Faculty, Students, admin staff, non faculty teaching staff, helping (guards, cleaners), system Admin should be able to sign up to the system.
    • Response Sequences: User goes to landing page URL. User clicks on sign up button. User is redirected to the create new profile form. User clicks sign up and is redirected to the homepage. System stores the data given by user for creating report.

    3. Create Report

    • Description and Priority: User should be able to create report.
    • Response Sequences: User goes to landing page URL. User is shown a form containing email, password and phone number fields along with forgot password button. User goes to create report option and create a report. System stores the report in the database. Report can be resolved.

    4. Edit comment

    • Description and Priority: User can edit a comment.
    • Response Sequences: Users goes to landing page URL. Users shown a form containing email and password fields along with forgot password button. User clicks login and is redirected to the comment section. Faculty goes to edit comment option and create a new comment. System stores the edited comment in the database.

    5. Create audio comment

    • Description and Priority: User can create a audio comment in the report section
    • Response Sequences: User goes to landing page URL. User is shown the audio comment section and click on it. The system keep the audio and convert it with text.

    6. Speech recognition

    • Response Sequences: The system will take the audio comment and convert it to text conversation. System stores the conversation in database.



    5. Other Non-Functional Requirements

    1. Performance Requirements

    The system should be able to track users information and speech recognition and store the data in database.

    2. Safety Requirements
    • This system may face many threats.
    • This could be someone who discloses the data form the database where it is located, in the registration office.
    • Using false identification. (Fake ID)
    3. Security requirements
    • The system will be secured and will respect the user’s privacy. Speech Recognition should be done in client's device
    • Administrator requirements: Only the administrator has the rights to manage any changes in the system. Only the administrator has the rights to manage any changes in the stored data set.
    • On a consistent basis, the registration office should ensure that they have the right information and student’s images.
    • The faculty will have complete control and will require username and password to log into the system.
    • Student’s will require username and password to log into their individual account.
    • Error messages will be popped up, when invalid inputs are given in order to inform the user that the input provided is not accepted.
    4. Software Quality Attributes
    • Accurate and precise process must be performed by the system to avoid problems.
    • The system must be flexible. Should be easily modified.
    • Execution process must be fast and responsive.
    • The system should possess maintainability, should be able to cope up with any encountered problems. UML will be used in the development process.
    • Ensuring portability. The software is a web-based application and is built in javascript and PostgreSQL so it’s platform independent.
    5. Business Rules
  • Administrator Requirement: Have a complete control to the system. Will be able to edit, add/remove an user. Ensures valid information.
  • Faculty's Requirement: They can access to student information. The faculty member will be able to control the report. Faculty's job is to add, edit, update and delete any record.
  • Student's Requirement: Students will be able to log into the system and check their resources shared by their faculties.
